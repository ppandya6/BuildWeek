The problem

Why contamination matters

If related patients, specimens, slides, crops, or re-encoded images appear in both training and testing data, a model can be evaluated on information it has already seen. That inflates reported performance and undermines benchmarks, papers, and product claims.

Lineage leaks silently

A patient, specimen, or slide identifier can quietly cross the split even when filenames look completely unrelated.

Hashes aren't enough

The same decoded pixels can be stored as PNG and BMP with different byte hashes — ordinary checksums walk right past them.

Found too late

Contamination usually surfaces after evaluation. Slide-of-Life makes the evidence inspectable before the numbers are trusted.

The method

Three concepts, kept scientifically separate

Slide-of-Life never blurs a factual observation with a policy judgement, or a judgement with an automated fix. Each stage is deterministic and traceable.

1

Factual detection

Deterministic detectors emit evidence — lineage overlaps, exact byte duplicates, decoded-pixel duplicates, and similarity candidates — each with source and row provenance. No claims, just observations.

2

Policy evaluation

An explicit SplitPolicy marks every factual finding as allowed, a violation, or requiring review — with a deterministic, human-readable reason.

3

Repair proposal

An optional deterministic assignment responds only to eligible confirmed findings and always requires researcher review. A proposal is never proof that a split is correct.

Optional AI, off by default. GPT-5.6 can only propose semantic column mappings for unresolved manifest headers, using privacy-bounded aggregate statistics — never raw rows or images. It never creates findings, decides policy, or makes repair decisions, and requires explicit acceptance.

Get running

Quick start

The demo is fully synthetic, deterministic, and small — about two minutes. Python 3.11+ required. Pick your platform:

PowerShell
git clone https://github.com/ppandya6/Slide-of-Life.git
Set-Location Slide-of-Life
py -3.11 -m venv .venv
& .\.venv\Scripts\python.exe -m pip install --upgrade pip
& .\.venv\Scripts\python.exe -m pip install "slide-of-life==0.1.0a1"
& .\.venv\Scripts\python.exe scripts\generate_demo.py --force
& .\.venv\Scripts\slide-of-life.exe audit `
    --train examples\demo\generated\train_manifest.csv `
    --test examples\demo\generated\test_manifest.csv `
    --images examples\demo\generated\images `
    --schema-map examples\demo\schema-map.yaml `
    --output artifacts\demo-audit --repair --force
bash
git clone https://github.com/ppandya6/Slide-of-Life.git
cd Slide-of-Life
python3 -m venv .venv
.venv/bin/python -m pip install --upgrade pip
.venv/bin/python -m pip install "slide-of-life==0.1.0a1"
.venv/bin/python scripts/generate_demo.py --force
.venv/bin/slide-of-life audit \
  --train examples/demo/generated/train_manifest.csv \
  --test examples/demo/generated/test_manifest.csv \
  --images examples/demo/generated/images \
  --schema-map examples/demo/schema-map.yaml \
  --output artifacts/demo-audit --repair --force
bash
# Core install — no AI, no API key
python3 -m pip install "slide-of-life==0.1.0a1"
slide-of-life --version
slide-of-life --help

# Optional AI schema assistance (still off unless requested)
python3 -m pip install "slide-of-life[ai]==0.1.0a1"

The audit intentionally exits with code 2 because the synthetic data contains policy violations — that means it succeeded and wrote its reports. Then open artifacts/demo-audit/report.html, a self-contained report that needs no server or internet connection.
